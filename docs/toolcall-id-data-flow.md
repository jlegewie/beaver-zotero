# Tool Call ID Data Flow

This document describes how `tool_call_id` values flow through the Beaver frontend, from arrival to consumption. It covers all ingress paths, storage mechanisms, and the matching logic that ties tool calls to their results, agent actions, and pending approvals.

---

## 1. Type Definitions

### 1.1 Message Parts (react/agents/types.ts)

Three part types carry a `tool_call_id`:

```typescript
interface ToolCallPart {
    part_kind: 'tool-call';
    tool_name: string;
    args: string | Record<string, any> | null;
    tool_call_id: string;           // generated by the LLM provider
    provider_details?: Record<string, any>;
    progress?: string;              // updated by WSToolCallProgressEvent
}

interface ToolReturnPart {
    part_kind: 'tool-return';
    tool_name: string;
    content: any;
    tool_call_id: string;           // matches the ToolCallPart it responds to
    metadata?: Record<string, any>;
}

interface RetryPromptPart {
    part_kind: 'retry-prompt';
    tool_name: string;
    content: any;
    tool_call_id: string;           // matches the failed ToolCallPart
}
```

### 1.2 Agent Actions (react/agents/agentActions.ts)

```typescript
interface AgentAction {
    id: string;                     // unique action UUID (server-generated)
    run_id: string;
    toolcall_id?: string;           // links to the ToolCallPart that triggered this action
    action_type: ActionType;
    status: ActionStatus;
    proposed_data: Record<string, any>;
    result_data?: Record<string, any>;
    // ...
}
```

### 1.3 Pending Approvals (react/agents/agentActions.ts)

```typescript
interface PendingApproval {
    actionId: string;               // the AgentAction.id
    toolcallId: string;             // for UI matching to ToolCallPart
    actionType: AgentActionType;
    actionData: Record<string, any>;
    currentValue?: any;
}
```

## 2. Ingress Paths — How tool_call_ids Arrive

### 2.1 WebSocket Streaming (real-time during agent run)

All WS events are handled in `react/atoms/agentRunAtoms.ts` → `createWSCallbacks()`.

#### 2.1.1 `WSPartEvent` → ToolCallPart

**Protocol** (src/services/agentProtocol.ts):
```typescript
interface WSPartEvent {
    event: 'part';
    run_id: string;
    message_index: number;          // index into AgentRun.model_messages
    part_index: number;             // index into ModelResponse.parts
    part: TextPart | ThinkingPart | ToolCallPart;
}
```

**Handler** (`onPart`, ~line 740):
- Calls `updateRunWithPart()` which stores the `ToolCallPart` at `model_messages[message_index].parts[part_index]` inside the active run.
- The `tool_call_id` is embedded in the part and persisted as-is.

#### 2.1.2 `WSToolReturnEvent` → ToolReturnPart / RetryPromptPart

**Protocol**:
```typescript
interface WSToolReturnEvent {
    event: 'tool_return';
    run_id: string;
    message_index: number;
    part: ToolReturnPart | RetryPromptPart;
}
```

**Handler** (`onToolReturn`, ~line 770):
- Calls `updateRunWithToolReturn()` which appends the part to `model_messages[message_index].parts`.
- If `event.part.part_kind === 'tool-return'`, also calls `processToolReturnResults()`.
- Only for `part_kind === 'tool-return'`, scans `pendingApprovalsAtom` for a `PendingApproval` whose `toolcallId` matches `event.part.tool_call_id`, and removes the first match.
- If the part is a `RetryPromptPart`, no pending-approval cleanup happens.

#### 2.1.3 `WSToolCallProgressEvent`

**Protocol**:
```typescript
interface WSToolCallProgressEvent {
    event: 'tool_call_progress';
    run_id: string;
    tool_call_id: string;           // direct field (not wrapped in a part)
    progress: string;
}
```

**Handler** (`onToolCallProgress`, ~line 799):
- Calls `updateRunWithToolCallProgress()` which searches **all model messages** in the run for a `ToolCallPart` with matching `tool_call_id`, then updates its `progress` field.
- Within the active run, it updates the **first matching response message only**. If the same `tool_call_id` is reused later in the run, later duplicates will not receive progress updates.

#### 2.1.4 `WSAgentActionsEvent`

**Protocol**:
```typescript
interface WSAgentActionsEvent {
    event: 'agent_actions';
    run_id: string;
    actions: AgentAction[];         // each has toolcall_id field
}
```

**Handler** (`onAgentActions`, ~line 958):
- Deserializes via `toAgentAction()` (handles both `toolcall_id` and `toolcallId` casing).
- Upserts into `threadAgentActionsAtom` (inserts new or updates existing by `action.id`).

#### 2.1.5 `WSRunCompleteEvent`

**Protocol**:
```typescript
interface WSRunCompleteEvent {
    event: 'run_complete';
    run_id: string;
    agent_actions: AgentAction[] | null;    // may contain actions with toolcall_id
    // ...
}
```

**Handler** (`onRunComplete`, ~line 804):
- If `event.agent_actions` is non-empty, deserializes via `toAgentAction()` and **adds** (not upserts) to `threadAgentActionsAtom`.
- This is the final delivery of actions; earlier `agent_actions` events provide incremental updates during the run.
- Because this path uses append rather than upsert, actions already received via earlier `agent_actions` events can be duplicated if the same action IDs are included again in `event.agent_actions`.

#### 2.1.6 `WSDeferredApprovalRequest`

**Protocol**:
```typescript
interface WSDeferredApprovalRequest {
    event: 'deferred_approval_request';
    action_id: string;
    toolcall_id: string;            // for UI matching to the ToolCallPart
    action_type: AgentActionType;
    action_data: Record<string, any>;
    current_value?: any;
}
```

**Handler** (`onDeferredApprovalRequest`, ~line 1013):
- May auto-approve for `confirm_extraction` / `confirm_external_search` based on user prefs.
- For `confirm_extraction`, there is an additional prevalidation path: if the locally resolvable attachment count is within `included_free`, the frontend auto-approves without adding a pending approval.
- Otherwise, calls `addPendingApprovalAtom` which stores a `PendingApproval` keyed by `action_id` in the `pendingApprovalsAtom` map. The `toolcall_id` is stored as `PendingApproval.toolcallId`.

### 2.2 REST API — Thread Loading (react/atoms/threads.ts)

When loading a previously stored thread via `loadThreadAtom` (~line 222):

```typescript
const { runs, agent_actions } = await agentRunService.getThreadRuns(threadId, true);
```

- **Runs**: Each `AgentRun` contains `model_messages[]` which contain `ToolCallPart` (with `tool_call_id`) and `ToolReturnPart` / `RetryPromptPart` (with `tool_call_id`). These are stored directly — no transformation of tool_call_ids.
- **Agent Actions**: `AgentRunService.getThreadRuns()` deserializes raw backend actions via `toAgentAction()` before returning them to `loadThreadAtom`. `loadThreadAtom` then stores those normalized actions in `threadAgentActionsAtom`.

The REST endpoint is `GET /api/v1/agents/beaver/threads/${threadId}/runs?include_actions=true` (src/services/agentService.ts → `AgentRunService.getThreadRuns`).

### 2.3 Summary of Ingress Points

| Source | Event/Endpoint | Parts carrying tool_call_id |
|--------|---------------|---------------------------|
| WS | `WSPartEvent` | `ToolCallPart.tool_call_id` |
| WS | `WSToolReturnEvent` | `ToolReturnPart.tool_call_id`, `RetryPromptPart.tool_call_id` |
| WS | `WSToolCallProgressEvent` | `event.tool_call_id` (top-level) |
| WS | `WSAgentActionsEvent` | `AgentAction.toolcall_id` (in each action) |
| WS | `WSRunCompleteEvent` | `AgentAction.toolcall_id` (in `event.agent_actions[]`) |
| WS | `WSDeferredApprovalRequest` | `event.toolcall_id` |
| REST | `getThreadRuns` | `ToolCallPart`, `ToolReturnPart`, `RetryPromptPart` in `model_messages[]`, plus `AgentAction.toolcall_id` |

## 3. Storage — Where tool_call_ids Live in State

### 3.1 Run State (react/agents/atoms.ts)

```
activeRunAtom (AgentRun | null) ─── streaming run
threadRunsAtom (AgentRun[]) ─────── completed runs
allRunsAtom (derived) ───────────── [...threadRuns, activeRun]
```

Each `AgentRun.model_messages` is an array of `ModelMessage` (either `ModelRequest` or `ModelResponse`). Tool call IDs live inside the parts:
- `ModelResponse.parts` → `ToolCallPart[]` (where tool calls are issued)
- `ModelRequest.parts` → `ToolReturnPart[]` / `RetryPromptPart[]` (where results return)

### 3.2 Tool Results Map (react/agents/atoms.ts:68)

```typescript
export const toolResultsMapAtom = atom<Map<string, ToolReturnPart | RetryPromptPart>>((get) => {
    const runs = get(allRunsAtom);
    const map = new Map();
    for (const run of runs) {
        for (const msg of run.model_messages) {
            if (msg.kind === 'request') {
                for (const part of msg.parts) {
                    if (part.part_kind === 'tool-return' || part.part_kind === 'retry-prompt') {
                        map.set(part.tool_call_id, part);  // ⚠️ keyed by tool_call_id alone
                    }
                }
            }
        }
    }
    return map;
});
```

**Key**: `tool_call_id` alone (flat across all runs in the thread).
**Collision risk**: If two runs contain the same `tool_call_id`, the later one silently overwrites the earlier one.

### 3.3 Agent Actions by Tool Call (react/agents/agentActions.ts:383-404)

```typescript
const agentActionsByToolcallAtom = atom<Map<string, AgentAction[]>>((get) => {
    const actions = get(threadAgentActionsAtom);
    return groupActionsByToolcall(actions);  // groups by action.toolcall_id
});

const getAgentActionsByToolcallAtom = atom(
    (get) => (toolcallId: string, filter?) =>
        get(agentActionsByToolcallAtom).get(toolcallId)?.filter(filter) || []
);
```

**Key**: `toolcall_id` alone (flat across all runs).
**Collision risk**: Same as above — actions from different runs with the same `toolcall_id` are mixed together.

### 3.4 Agent Actions by Run (react/agents/agentActions.ts:410-429)

```typescript
const agentActionsByRunAtom = atom<Map<string, AgentAction[]>>(...);
const getAgentActionsByRunAtom = atom(
    (get) => (runId: string, filter?) => ...
);
```

**Key**: `run_id`. Used in `AgentActionsReview` for per-run filtering. Filters further by `toolcall_id === 'citations'` for citation-based imports.

### 3.5 Pending Approvals (react/agents/agentActions.ts:743)

```typescript
const pendingApprovalsAtom = atom<Map<string, PendingApproval>>(new Map());
```

**Key**: `action_id` (globally unique).
**Lookup by tool_call_id**: `getPendingApprovalForToolcallAtom` iterates the map and returns the first `PendingApproval` where `pending.toolcallId === toolcallId`. Returns at most one match.

Pending approvals are also lifecycle-sensitive:
- Cleared on `newThreadAtom`
- Cleared on `loadThreadAtom`
- Cleared on `closeWSConnectionAtom` (stop/cancel path)
- Cleared individually when the user responds, or when a matching `ToolReturnPart` arrives

### 3.6 UI State Atoms (react/atoms/messageUIState.ts)

| Atom | Key format | Used in |
|------|-----------|---------|
| `toolExpandedAtom` | `${runId}:${responseIndex}:${toolcallId}` | `ToolCallPartView` |
| `toolExpandedAtom` | `${runId}:${toolcallId}` | `AgentActionView` |
| `agentActionItemTitlesAtom` | `${toolcallId}` | `AgentActionView` |
| `annotationPanelStateAtom` | `${toolCallId}` | `AnnotationToolCallView` |
| `annotationBusyAtom` | `${toolCallId}` | `AnnotationToolCallView` |
| `annotationAttachmentTitlesAtom` | `${toolCallId}` | `AnnotationToolCallView` |

Note the **inconsistency** in key formats: `ToolCallPartView` uses a composite key with `responseIndex`, while `AgentActionView` uses `${runId}:${toolcallId}` (no responseIndex), and `AnnotationToolCallView` uses bare `toolCallId`.

These atoms are also tied into scroll behavior:
- `ThreadView` watches `toolExpandedAtom`
- `ThreadView` watches `annotationPanelStateAtom`

When these keyed states change, `ThreadView` re-evaluates whether the thread is considered scrolled away from the bottom. So collisions here can affect scroll behavior, not just expansion state.

## 4. Matching Logic — How tool_call_ids Connect Things

### 4.1 ToolCallPart ↔ ToolReturnPart (result matching)

**Where**: `ToolCallPartView` (react/components/agentRuns/ToolCallPartView.tsx:170)
```typescript
const result = resultsMap.get(part.tool_call_id);
```

Uses `toolResultsMapAtom` (keyed by `tool_call_id` alone). The `ToolCallPart` from a `ModelResponse` is matched to a `ToolReturnPart` from a subsequent `ModelRequest` by their shared `tool_call_id`.

### 4.2 ToolCallPart ↔ AgentAction (action matching)

**Where**: `ToolCallPartView` (line 182), `AgentActionView` (line 252), and `AnnotationToolCallView` (line 365)
```typescript
// ToolCallPartView — checks if any actions exist for this tool call
const agentActions = getAgentActionsByToolcall(part.tool_call_id);

// AgentActionView — gets all actions (no additional filter)
const actions = getAgentActionsByToolcall(toolcallId);
const annotations = getAgentActionsByToolcall(toolCallId, isAnnotationAgentAction);
```

All three use bare `toolcall_id` with **no run or response scoping**.

Important nuance: not every tool call with matching actions renders `AgentActionView`.

- Annotation tools (`add_highlight_annotations`, `add_note_annotations`, `add_annotations`) are routed to `AnnotationToolCallView`.
- `AgentActionView` is only used for:
  - `edit_metadata`
  - `create_collection`
  - `organize_items`
  - `create_items`
  - pending confirm flows for `extract` / `external_search`
- Citation-derived imports (`toolcall_id === 'citations'`) are rendered separately in `AgentActionsReview`, not beneath a `ToolCallPart`.

### 4.3 ToolCallPart ↔ PendingApproval (approval matching)

**Where**: `ToolCallPartView` (line 177)
```typescript
const pendingApproval = getPendingApproval(part.tool_call_id);
```

Uses `getPendingApprovalForToolcallAtom` which does a linear scan of `pendingApprovalsAtom` values, matching on `pending.toolcallId === toolcallId`.

### 4.4 ToolCallPart ↔ Progress (progress matching)

**Where**: `updateRunWithToolCallProgress()` (react/agents/atoms.ts:255)
```typescript
const toolCallPart = message.parts.find(
    part => part.part_kind === 'tool-call' && part.tool_call_id === event.tool_call_id
);
```

Searches all `ModelResponse` messages in the **active run** for a `ToolCallPart` with matching `tool_call_id`. Since this only operates on the active run, scoping is implicitly per-run.

Within that active run, however, it updates the **first matching response message only**. If the same `tool_call_id` is reused later in the run, later duplicates will not receive progress updates.

### 4.5 PendingApproval cleanup on successful tool return

**Where**: `onToolReturn` handler (react/atoms/agentRunAtoms.ts:786-795)
```typescript
const toolCallId = event.part.tool_call_id;
const pendingMap = store.get(pendingApprovalsAtom);
for (const [actionId, pending] of pendingMap.entries()) {
    if (pending.toolcallId === toolCallId) {
        set(removePendingApprovalAtom, actionId);
        break;
    }
}
```

When a `ToolReturnPart` arrives, any pending approval for that `tool_call_id` is removed. This prevents stale approval dialogs from lingering after the backend has already processed the tool call.

This cleanup does **not** happen for `RetryPromptPart`.

### 4.6 Special: `toolcall_id === 'citations'`

**Where**: `AgentActionsReview` (react/components/agentRuns/AgentActionsReview.tsx:29)
```typescript
const createItemActions = getAgentActionsByRun(
    run.id,
    (action) => isCreateItemAgentAction(action) && action.toolcall_id === 'citations'
);
```

The backend assigns the synthetic `toolcall_id` value `'citations'` to `create_item` actions that come from citation extraction (not from a regular tool call). This is a semantic marker, not a reference to an actual `ToolCallPart`.

### 4.7 Other frontend consumers of `tool_call_id`

Beyond the direct tool-call views, `tool_call_id` is also consumed by:

- **`AgentRunView`**: iterates all `ToolCallPart`s and calls `getToolCallStatus(part.tool_call_id, ...)` to determine whether any tool call is still in progress. This affects whether the run-level spinner/status indicator is shown.
- **`threadContent.ts`**: when exporting/copying thread content, `getToolCallDetails()` uses `part.tool_call_id` to look up the matching result in `toolResultsMapAtom` and include result counts in the generated text.

## 5. Component Tree and Prop Flow

```
ModelMessagesView
  messages.map((message, index) => {
    if (message.kind === 'response') {
      <ModelResponseView
        responseIndex={index}           ← index into model_messages array
        runId={runId}
      >
        toolCallParts.map(part => {
          // Annotation tools:
          <AnnotationToolCallView
            key={`tool-${part.tool_call_id}`}       ← React key
            part={part}                              ← contains tool_call_id
            runId={runId}
          />

          // Other tools:
          <ToolCallPartView
            key={`tool-${responseIndex}-${part.tool_call_id}`}  ← React key
            part={part}                              ← contains tool_call_id
            runId={runId}
            responseIndex={responseIndex}
          >
            // If tool has agent actions:
            <AgentActionView
              toolcallId={part.tool_call_id}
              runId={runId}

            />
          </ToolCallPartView>
        })
      </ModelResponseView>
    }
  })
```

**Important**: `responseIndex` is passed to `ToolCallPartView` but is **not** forwarded to `AgentActionView`. It is only used by `ToolCallPartView` for its own expansion key (`${runId}:${responseIndex}:${tool_call_id}`).

Also note that this tree is conditional:
- Annotation tools branch to `AnnotationToolCallView`
- Only a subset of non-annotation tools ever render `AgentActionView`
- Citation import actions with `toolcall_id === 'citations'` are rendered separately by `AgentActionsReview`, outside this `ToolCallPart` subtree

## 6. Relationship: `message_index` vs `responseIndex`

These are related but not identical:

- **`message_index`** (WS protocol): Index into `AgentRun.model_messages[]`, which is a flat array of alternating `ModelRequest` and `ModelResponse` messages. Used by `updateRunWithPart()` and `updateRunWithToolReturn()` to place parts at the correct position.

- **`responseIndex`** (component prop): Index into the `model_messages[]` array as iterated by `ModelMessagesView.messages.map()`. Since `ModelMessagesView` iterates **all** messages but only renders `kind === 'response'` messages, `responseIndex` equals the raw array index. So `responseIndex` and `message_index` are the same value for response messages.

## 7. Collision Risks and Inconsistencies

### 7.1 Cross-run tool_call_id collisions

If two different runs in the same thread happen to use the same `tool_call_id`:

| Affected component | Key used | Behavior |
|---|---|---|
| `toolResultsMapAtom` | `tool_call_id` alone | Later run's result **overwrites** earlier run's result |
| `agentActionsByToolcallAtom` | `toolcall_id` alone | Actions from both runs are **mixed** into the same bucket |
| `getPendingApprovalForToolcallAtom` | `toolcallId` search | Would return the **first match** (could be from wrong run) |
| `ToolCallPartView` expansion | `${runId}:${responseIndex}:${tool_call_id}` | **No collision** (scoped by runId) |
| `AgentActionView` expansion | `${runId}:${toolcallId}` | **No collision** (scoped by runId) |
| `WSRunCompleteEvent` action ingestion | append by array order | Previously streamed actions can be **duplicated** in `threadAgentActionsAtom` |

**Practical risk**: Depends on whether the backend/LLM provider generates globally unique tool_call_ids (e.g., UUIDs) or sequential ones (e.g., `call_1`, `call_2`). If sequential, collisions across runs are almost certain.

### 7.2 Within-run tool_call_id reuse

If the same `tool_call_id` appears in multiple responses within a single run (e.g., retried tool calls):

| Affected component | Key used | Behavior |
|---|---|---|
| `toolResultsMapAtom` | `tool_call_id` alone | Later result **overwrites** earlier |
| `agentActionsByToolcallAtom` | `toolcall_id` alone | Both responses' actions are mixed |
| `updateRunWithToolCallProgress()` | first matching response only | Later duplicate tool calls may **never show progress** |
| `AgentActionView` expansion | `${runId}:${toolcallId}` | Same expansion key → **shared state** between both instances |
| `ToolCallPartView` expansion | `${runId}:${responseIndex}:${tool_call_id}` | **No collision** (responseIndex differs) |
| `ToolCallPartView` React key | `tool-${responseIndex}-${tool_call_id}` | **No collision** |
| `AnnotationToolCallView` React key | `tool-${tool_call_id}` | **Collision** (same React key) |

### 7.3 Key format inconsistencies

The expansion key format varies across components:

| Component | Expansion key format |
|---|---|
| `ToolCallPartView` | `${runId}:${responseIndex}:${tool_call_id}` |
| `AgentActionView` | `${runId}:${toolcallId}` |
| `AnnotationToolCallView` | `${toolCallId}` (bare) |

### 7.4 Action lookup inconsistency

`AgentActionView` calls `getAgentActionsByToolcall(toolcallId)` with **no filter**, meaning it receives actions across all runs. However, it only has `runId` available (not `responseIndex`), so it cannot scope further even if it wanted to.

By contrast, `ToolCallPartView` also calls `getAgentActionsByToolcall(part.tool_call_id)` without filtering — but this is only used to check `hasAgentAction` (boolean), not to render specific actions.

### 7.5 State reset and lifecycle gaps

Several `tool_call_id`-keyed structures are intentionally ephemeral and are cleared when switching context:

| State | Cleared by |
|---|---|
| `pendingApprovalsAtom` | `newThreadAtom`, `loadThreadAtom`, `closeWSConnectionAtom`, per-action removal |
| `toolExpandedAtom` | `resetMessageUIStateAtom` |
| `annotationPanelStateAtom` | `resetMessageUIStateAtom` |
| `annotationBusyAtom` | `resetMessageUIStateAtom` |
| `annotationAttachmentTitlesAtom` | `resetMessageUIStateAtom` |
| `agentActionItemTitlesAtom` | `resetMessageUIStateAtom` |

This matters because the frontend does **not** persist these `tool_call_id`-derived UI states across thread resets or new-thread creation.
